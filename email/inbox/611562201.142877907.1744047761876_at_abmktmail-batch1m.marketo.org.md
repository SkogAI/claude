We are writing to inform you of high-severity security updates to our products that require your immediate attention.
Summary
High-severity vulnerabilities have been identified in all Apollo Router versions up to and including v1.61.1 and v2.1.0 and all Apollo Gateway versions up to and including v2.10.0. These security issues could potentially allow unauthorized attackers to bypass system-enforced limits or exploit performance weaknesses. Security advisories are linked below.
Today we have released Router v1.61.2, v2.1.1, and Gateway v2.10.1 to address the three vulnerabilities affecting GraphQL validation, query planning, and the operation limits plugin. We strongly recommend upgrading to these new versions as soon as possible. 
Impact
If exploited, these vulnerabilities could have the following effects:


	- Apollo Router and/or Apollo Gateway processes rendered inoperable, resulting in an outage.
	- Excessive resource consumption of Apollo Router and/or Apollo Gateway processes.


Required Actions
Please take the following actions immediately:


	- Update all instances of Apollo Router to >=v1.61.2 or >=v2.1.1
	- Update all instances of Apollo Gateway to >=v2.10.1


Updated Release Information


	- The security release details for customers running v1.x of Apollo Router is available here: Apollo Router v1.61.2 <https://github.com/apollographql/router/releases/tag/v1.61.2>
	- The security release details for customers running v2.x of Apollo Router is available here: Apollo Router v2.1.1 <https://github.com/apollographql/router/releases/tag/v2.1.1>
	- The security release details for customers running all instances of Apollo Gateway is available here: Apollo Gateway v2.10.1 <https://github.com/apollographql/federation/releases/tag/%40apollo%2Fgateway%402.10.1>


The corresponding releases are available on all of our official distribution channels, including Docker, Helm, npm, GitHub Releases, and Crates.io.
Mitigation Details
A detailed description of the underlying issues is provided below.
1. Named Fragment Processing Vulnerability
Component: GraphQL Validation
Impact: DoS due to excessive resource consumption.
 Details: Named fragments when used with certain query patterns could lead to exponential resource usage.
Fix/Mitigation: The validation logic has been updated to process each named fragment only once, preventing redundant traversal.


Component: Query Planner
 Impact: Potential DoS or excessive resource consumption.
 Details: Named fragments were being expanded once per fragment spread, leading to exponential resource usage when deeply nested or reused fragments were involved.
Fix/Mitigation: To address this, we are introducing a new metric for Query Fragment Expansion Limits:




	- This new metric will compute the number of selections that would be created if fragments were expanded.
	- This metric will be checked against a limit to prevent excessive computation.




2. Operation Limits Plugin Vulnerability
Component: Operation Limits Plugin
 Impact: Limit bypass due to counter overflow.
 Details: The plugin tracks operation limit counters using unsigned 32-bit integers. When integer maximums were exceeded, counters wrapped around to 0, allowing large operations to bypass configured thresholds.
Fix/Mitigation: Logic now ensures counter overflow is correctly handled, and operation limit counters exceeding u32::MAX no longer reset to 0.

3. Query Planner Optimization Vulnerability


Component: Query Planner
 Impact: Potential DoS or excessive resource consumption.
 Details: Query planner has an internal optimization that greatly speeds up planning when applicable. The fragment spread issue can be used to create medium-sized operations where the optimization is never applicable, resulting in significant query planning time. Since query planning has no timeout, a small number of malicious queries can exhaust the query planning thread pool.
Fix/Mitigation: To address this, we are introducing a new metric for Query Optimization Limits:




	- This new metric will approximate the number of selections that cannot be skipped by the existing optimization.
	- This metric will be checked against a limit to prevent excessive computation.




Given the complexity of query planning optimizations, we will refine these solutions as needed based on real-world performance and accuracy tests.
