<h2>What makes Opus 4.5 an infinite coding machine? </h2>
<p>Dan calls Opus 4.5 the <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2V2ZXJ5LnRvL2NoYWluLW9mLXRob3VnaHQvb3B1cy00LTUtY29sbGFwc2VkLXNpeC1tb250aHMtb2YtZGV2ZWxvcG1lbnQtd29yay1pbnRvLW9uZS13ZWVrIiwicG9zaXRpb24iOjE1fQ==" rel="noopener noreferrer" target="_blank">infinite coding machine</a></u>. Previous models were great for demos but would peter out when you tried to build something real—three steps in, they’d start hallucinating fixes and making the same errors over and over again. Opus doesn’t hit that wall.</p>
<p>The shift has implications on two fronts: how you think about software (coding philosophy) and how you structure that software (code architecture).</p>
<h3><strong>Coding philosophy: Depth and delegation</strong></h3>
<p>Dan started building his reading app with other frontier models before Opus 4.5 arrived. He’d make progress, then hit the error wall. When he switched to Opus, the pattern changed. “It kept going,” he said during the camp. “I kept adding features and adding features.”</p>
<p>One of the reasons Opus 4.5 can do this is because it can fix problems in code you didn’t even write. Usually, if something breaks, you can only fix the issue in your own code. Anything else, you would have to become an expert in that tool to fix it. For example, if your app uses React, a framework for building web apps, you are also dependent on specific code that React uses to display content on screen, which in turn depends on the interface that lets code communicate with a browser like Chrome or Safari. Now, Opus 4.5 can trace a bug through all three layers and fix it.</p>
<p>Opus 4.5 also lets you work on multiple things in parallel, by delegating multiple tasks to the model at once. This increases the speed at which you can build.</p>
<h3><strong>Code architecture: Agent-native apps</strong></h3>
<p>The second shift is architectural. Opus changes how apps can be structured.</p>
<p>Traditionally, you code each feature as a step-by-step recipe. Every action is precisely defined: if user clicks button → fetch data → format results → display. </p>
<p>With Opus, you can build what Dan calls “agent-native” apps. Instead of coding recipes, you build a general-purpose agent (an AI that can use tools and follow instructions, <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2V2ZXJ5LnRvL2NoYWluLW9mLXRob3VnaHQvb3B1cy00LTUtY29sbGFwc2VkLXNpeC1tb250aHMtb2YtZGV2ZWxvcG1lbnQtd29yay1pbnRvLW9uZS13ZWVrIiwicG9zaXRpb24iOjE2fQ==" rel="noopener noreferrer" target="_blank">similar to a chef</a></u> that can follow recipes and cook) with access to tools, then give it prompts describing outcomes. The agent figures out the steps.</p>
<p>Dan’s reading app includes a feature that determines your “reading profile” from books and articles you have screenshotted on your phone’s camera roll. To code this traditionally, you would have had to write an algorithm to efficiently scan through a gigantic photo library, identify photos that are likely to contain text, convert them from images to text, collect them, and summarize them into a coherent, interesting synthesis—a complex process.  Now, when you use the reading profile feature, the app calls the agent with a prompt. The agent scans your camera roll, identifies books, analyzes your taste, and writes it up—no feature code required.</p>
<p>Features are faster to build because you’re writing prompts instead of code, and they’re faster to iterate because you edit the prompt instead of hunting down the exact line of code that broke.</p>
<p>The tradeoffs are significant. Agent-native apps are more expensive—every feature costs more money to run because they consume costly compute resources, instead of just executing pre-written code—and slower, because the agent has to think. But model costs are dropping, and you don’t have to go all in. Use agent-native architecture for exploration, then write common workflows into traditional code as patterns emerge.</p>
<h2><strong>What a senior engineer can do with it</strong></h2>
<p>The philosophy and architecture shifts unlock workflows that weren’t possible before. Kieran demonstrated three patterns that show what Opus 4.5 looks like in daily use.</p>
<h3><strong>Computer use for testing</strong></h3>
<p>Previous models could technically control browsers, but they’d get confused, click the wrong things, or lose track. Opus stays coherent through complex multi-step tasks, which makes computer use viable for tasks like regression testing (checking that new code doesn’t break existing features) and user interface validation that would previously require manually clicking through your entire app.</p>
<p>Kieran uses <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL3BsYXl3cmlnaHQuZGV2LyIsInBvc2l0aW9uIjoxN30=" rel="noopener noreferrer" target="_blank">Playwright</a></u> for end-to-end testing (checking that features work from start to finish in a real browser). During the demo, he showed how he would test a new feature using Opus 4.5. Instead of manually clicking through each link and button in the app to make sure nothing was broken, he told Opus: “Use Playwright to test this feature. Click through everything and make sure nothing is broken.”</p>
<p>The model booted up Chrome, navigated the interface, found a bug, went back to the code, fixed it, and continued testing. All on its own.</p>
<p>He’s also automated before/after screenshots for pull requests (drafts of code changes submitted for review). Every time he uses the <code>/work</code> command (which tells the AI to start building), Playwright captures screenshots of the user interface before and after the change, then adds them to the pull request. Code reviewers can see exactly what changed visually without running the code themselves.</p>
<p>AI controlling your computer used to be a flashy trick. Now, it’s something you can rely on for real work.</p>
<h3><strong>Parallel delegation</strong></h3>
<p>Within days of getting Opus 4.5 access, Kieran was building 10 things at the same time. He confessed to being “agent addicted”—sneaking away from Thanksgiving dinner to start new prompts, shutting down his devices at night to force himself to stop.</p>
<p>Previous models would bleed context between tasks—they would take some information or instructions you used in one project and apply it in another, or lose track of details. Opus holds the thread on all of them, so the amount of work you are capable of is as much as you can hold in your head at one time. </p>
<p>One of Kieran’s projects was finishing an abandoned feature for <strong><u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2NvcmEuY29tcHV0ZXIiLCJwb3NpdGlvbiI6MTh9" rel="noopener noreferrer" target="_blank">Cora</a></u></strong> that would detect warning signs that a user is about to stop using the product. He’d spent days trying to build it with previous models. They’d write half the code and get stuck, or they’d stop to ask clarifying questions about every small decision—should the button be on the left or right, what color should the error message be, how many seconds before timing out. Opus 4.5 finished it in 30 minutes. When it hit a small decision, it made a reasonable choice and kept building instead of stopping to ask.</p>
<p>This pattern works when you’re launching several features at once, testing multiple approaches at the same time, or doing maintenance work across different parts of a codebase.</p>
<h3><strong>Meta-level work</strong></h3>
<p>Kieran was working on Every Board, a feedback tool the team is building internally, when he realized the <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2V2ZXJ5LnRvL3NvdXJjZS1jb2RlL215LWFpLWhhZC1hbHJlYWR5LWZpeGVkLXRoZS1jb2RlLWJlZm9yZS1pLXNhdy1pdCIsInBvc2l0aW9uIjoxOX0=" rel="noopener noreferrer" target="_blank">compound engineering</a></u> plugin needed a fourth research agent for design work. Usually, that would require him to stop what he was doing, open up the new tool’s code in a different place, make changes, and go back to his original project—by which time he would have forgotten where he was in coding. Instead, he created a command called <code>/modify-plugin</code> that let the AI work on both things at once: the Every Board project and making changes to the compound engineering framework—the set of instructions that Kieran has built to define the <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2V2ZXJ5LnRvL3NvdXJjZS1jb2RlL2hvdy1pLXVzZS1jbGF1ZGUtY29kZS10by1zaGlwLWxpa2UtYS10ZWFtLW9mLWZpdmUiLCJwb3NpdGlvbiI6MjB9" rel="noopener noreferrer" target="_blank">four steps</a></u> that make up his AI-powered coding workflow.. </p>
<p>“This is 4.5,” Kieran said. “This kind of work would have never been possible before without it becoming a complete mess.”</p>
<p>The model maintained what it knew about the design problems Kieran was hitting with the Every Board project, then switched to the codebase for the compound engineering workflow, which contains all of the rules and behaviors that make compound engineering work. It analyzed how the research agents were defined there and added a new agent that fit the pattern.</p>
<p>With Opus, you work on your project while modifying the tools you’re using to build that project simultaneously. This is particularly useful when you’re building tools, plugins, or systems that all need to work together. </p>
<h2><strong>The Q&amp;A</strong></h2>
<p>Over the second half of the camp, Dan and Kieran fielded questions about their workflows. Here are the highlights.</p>
<h3><strong>Planning and compound engineering</strong></h3>
<h5><strong>Q: Is planning still necessary with Opus?</strong></h5>
<p>Kieran: For exploratory work or personal projects, Opus is good enough that you can just start building. But if you’re working on a team or building something complex, planning is more valuable than ever because Opus can execute much deeper, more sophisticated plans.</p>
<h5><strong>Q: When do you use the compound engineering workflow versus just vibing?</strong></h5>
<p>Kieran: If I already know exactly what I want to build and it’s production work, I use the full workflow—plan, delegate, assess, codify. For personal projects or exploration, I skip it.</p>
<h3><strong>Computer use and testing</strong></h3>
<h5><strong>Q: How do I get Playwright working with Claude Code?</strong></h5>
<p>Kieran: Install the <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2dpdGh1Yi5jb20vRXZlcnlJbmMvY29tcG91bmQtZW5naW5lZXJpbmctcGx1Z2luIiwicG9zaXRpb24iOjIxfQ==" rel="noopener noreferrer" target="_blank">compound engineering plugin</a></u>—it’s two lines in your terminal and Playwright is bundled. Then just ask Claude to test something.</p>
<h5><strong>Q: Can Opus replace manual testing entirely?</strong></h5>
<p>Kieran: Not yet. It’s way easier to add unit tests now, and end-to-end testing with Playwright works well for web apps. But some manual testing is still needed.</p>
<h3><strong>Parallel work and delegation</strong></h3>
<h5><strong>Q: How many projects can you realistically work on at once?</strong></h5>
<p>Kieran: However many your brain can handle. That’s the actual bottleneck now. The AI can manage more parallel work than you can keep track of.</p>
<p>Dan: Three to five substantial things feels manageable for me. Beyond that, I start losing track.</p>
<h3><strong>Working style and fatigue</strong></h3>
<h5><strong>Q: Do you hit a fatigue wall working with multiple AI agents?</strong></h5>
<p>Kieran: I use the <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL3d3dy50b2RvaXN0LmNvbS9wcm9kdWN0aXZpdHktbWV0aG9kcy9wb21vZG9yby10ZWNobmlxdWUiLCJwb3NpdGlvbiI6MjJ9" rel="noopener noreferrer" target="_blank">Pomodoro method</a></u> and bunch certain types of work together. Code reviews for an hour, then research for an hour. The fatigue isn’t from the volume of work—the AI handles that. It’s from switching between different ways of thinking about what you’re asking the AI to do.</p>
<h3><strong>Agent native architecture</strong></h3>
<h5><strong>Q: How expensive is the reading app to run?</strong></h5>
<p>Dan: Very expensive right now. Every feature call costs money. But model costs are dropping fast, and you don’t have to keep it agent-native forever. You can harden common patterns into regular code once you figure out what users actually do.</p>
<h5><strong>Q: Is this architecture suitable for production?</strong></h5>
<p>Dan: It’s production right now. I use this app every day. Kieran’s kids use the story generator. The tradeoff is cost and speed, but if those work for your use case, it’s completely viable.</p>
<h3><strong>Skills and context management</strong></h3>
<h5><strong>Q: What are skills and how do you use them?</strong></h5>
<p>Kieran: <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiSG93IEV2ZXJ5IElzIEhhcm5lc3NpbmcgdGhlIFdvcmxkLWNoYW5naW5nIFNoaWZ0IG9mIE9wdXMgNC41ICIsInBvc3RfaWQiOjM4NjIsInBvc3RfdHlwZSI6InBvc3QiLCJ1cmwiOiJodHRwczovL2V2ZXJ5LnRvL3ZpYmUtY2hlY2svdmliZS1jaGVjay1jbGF1ZGUtc2tpbGxzLW5lZWQtYS1zaGFyZS1idXR0b24iLCJwb3NpdGlvbiI6MjN9" rel="noopener noreferrer" target="_blank">Skills</a></u> are bundles of knowledge you can load to perform dedicated tasks. Instead of cramming everything into your Claude.md file, you create skills for specific technologies that trigger only when you need them. When you’re doing design work, load the design skill. It keeps your context window clean and gives Claude deeper knowledge exactly when it needs it.</p>
