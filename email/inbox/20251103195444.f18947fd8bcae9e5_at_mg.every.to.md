<h2><strong>What Skills do</strong></h2>
<p>Think of Skills as mini-experts that you can summon when neededвҖ”or Claude can invoke for you, if you write the prompt that way. You create a folder containing a SKILL.md file with instructions, optional Python scripts, and reference materials. Zip it, upload it, and Claude can use those instructions to work the way you work.</p>
<p>Need Claude to match your editorial voice? Build a skill. Want it to generate PowerPoints according to your companyвҖҷs house style? Package those rules as a skill. Have a complex workflow requiring multiple data files and custom codescripts? Skills can bundle all of that together.</p>
<p>The technical innovation is called вҖңprogressive disclosure.вҖқ Claude doesnвҖҷt load all your skills into its conversation space at once. Instead, it sees skill names and descriptions first, like scanning a bookshelf. When you ask Claude to do somethingвҖ”like check a post for alignment with EveryвҖҷs style guideвҖ”it identifies relevant skills and loads only what it needs. So you can package unlimited institutional knowledge without eating up the space Claude uses to track your conversation.</p>
<p>If youвҖҷve used Claude CodeвҖҷs <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiVmliZSBDaGVjazogQ2xhdWRlIFNraWxscyBOZWVkIGEg4oCYU2hhcmXigJkgQnV0dG9uIiwicG9zdF9pZCI6MzgxMSwicG9zdF90eXBlIjoicG9zdCIsInVybCI6Imh0dHBzOi8vZXZlcnkudG8vdmliZS1jaGVjay92aWJlLWNoZWNrLWNsYXVkZS1zLW5ldy1hZ2VudHMtYXJlLWNvbmZ1c2luZy1hcy1oZWxsIiwicG9zaXRpb24iOjE2fQ==" rel="noopener noreferrer" target="_blank">subagents</a></u>, Skills work on a similar principleвҖ”specialized capabilities you invoke when needed. The difference is that Skills work across Claude interfaces, including its web interface and API, rather than just in the coding environment.</p>
<p>The key to creating skills is understanding what you need to do versus what Claude can do for you. You provide the specificationsвҖ”tell Claude what outcome youвҖҷd like (вҖңI need a skill that checks drafts for AI writing patternsвҖқ), how you want it to behave, what a good output looks like, and any specific rules or preferences. Claude translates that into a skill file with proper formatting and structure. You can write skills yourself manually from scratch, of course. The question is, why would you? (I donвҖҷt.) </p>
<p>WhatвҖҷs the difference between Skills and Projects, dedicated workspaces where you can set up custom instructions and add documents? Projects apply their instructions automatically to everything in that workspace. Skills are on-demand capabilities that Claude loads anywhere across your Claude instanceвҖ”but only when the skill is either invoked by prompting it with вҖңuse this skillвҖқ or Claude calls it automatically, depending on how the instructions are written. My <u><a href="https://every.to/emails/click/f91b1e00930c1c7802951fb76da18da9d1d86023a9ec7b653e47ca1b5d584dbf/eyJzdWJqZWN0IjoiVmliZSBDaGVjazogQ2xhdWRlIFNraWxscyBOZWVkIGEg4oCYU2hhcmXigJkgQnV0dG9uIiwicG9zdF9pZCI6MzgxMSwicG9zdF90eXBlIjoicG9zdCIsInVybCI6Imh0dHBzOi8vZXZlcnkudG8vd29ya2luZy1vdmVydGltZS9pLWZlZC1teS1lc3NheXMtdG8tY2hhdGdwdC11bnRpbC1pdC1sZWFybmVkLW15LXZvaWNlIiwicG9zaXRpb24iOjE3fQ==" rel="noopener noreferrer" target="_blank">Working Overtime Project</a></u> contains my writing voice and style. <p>The experiment revealed something important about Skills architecture. Even Skills without code can be powerfulвҖ”theyвҖҷre essentially subagents for Claude, packaging detailed instructions that get invoked on-demand. NityeshвҖҷs PowerPoint design skill contained no code, just comprehensive design principles. Claude called upon those principles when needed, used them to create a blueprint, and passed that blueprint to the code-running PowerPoint skill to generate the file.</p>
<p>After a weekend of exploration, NityeshвҖҷs verdict is that the Skills feature is more powerful than Claude CodeвҖҷs subagents because a skill can bundle custom instructions with multiple data files and code while working inside the main chat context, rather than separating everything out into different agents.</p>

<h2><strong>The skill issue nobody mentioned</strong></h2>

But it overlooked the specifics of EveryвҖҷs style guide, because it hadnвҖҷt invoked the skill to do the editвҖ”it had just fallen back on its background knowledge about what a вҖңstyle checkвҖқ is. </p>

<h2><strong>Your team can't use your best SkillsвҖ”yet</strong></h2>
<p>Skills work beautifully for the AI-fluent individual. If you're already immersed in AI workflows, building Skills to enhance your personal productivity is worth the learning curve. The work that goes into setup pays off when you're managing your own tools.</p>
<p>But when I think about rolling this out across EveryвҖ”where Skills could really shineвҖ”I can think of three features that, if added, would take it from personal productivity tool to essential organizational infrastructure:</p>
<p><strong>Sharing infrastructure.</strong> Right now, there's no "share with team" button or central library where colleagues can browse and enable skills. If I want team members to use a skill, I have to notify the team, send the files, and hope the upload goes smoothly for everyone. The good news: Anthropic is working on simplified skill creation workflows and enterprise-wide deployment capabilities to make distributing skills across teams easier.</p>
<p><strong>Iteration infrastructure.</strong> Every change requires downloading the skill file, editing it locally, re-zipping it, and re-uploading. There's no way to test changes within Claude before deploying, or make quick tweaks and see results immediately. For a skill you're actively refiningвҖ”like my AI checker, which I kept thinking of things to add toвҖ”this download-edit-upload loop gets tedious. It's manageable when building one or two skills for yourself, but imagine maintaining a library of 20 skills across a team.</p>
<p><strong>Description quality guidance.</strong> As I learned the hard way, writing skill descriptions that Claude can parse correctly is its own skill. If I struggle with thisвҖ”and I write about AI for a livingвҖ”rolling this out to less AI-experienced team members will require either extensive documentation or someone reviewing skills before deployment.</p>
<p>Making these systems useful at scale requires someone making it their job to experiment, develop systems, educate, and enable others. For Every, that person is me. For a 500-person company with department-specific Skills libraries, the operational overhead adds up quickly.</p>
<h2><strong>The Reach Test</strong></h2>
<p><strong>Katie:</strong> For my individual work, Skills are already essential. My AI-check, ELI5, and Every-style Skills are daily driversвҖ”now that I've learned to write better descriptions. The learning curve around skill descriptions was steeper than I expected, but itвҖҷs worth it for the mental overhead it eliminates. At one point, Skills mysteriously disappeared from my account for a few days, and I immediately felt the difference (thankfully it came back, though I still donвҖҷt know why). That's the clearest signal I can give: I missed it when it was gone.</p>
<p>What Skills represent matters more than the current challenges. ItвҖҷs a way to formalize knowledge so Claude enforces the same standards and performs the same behaviors every time. That's already a big deal for individual productivity. For organizational deployment, we're waiting on sharing infrastructure and better onboarding around description quality. Anthropic says theyвҖҷre working on it. Until then, I'll keep building Skills for my own workflow.</p>
<p><strong>Nityesh:</strong> I love Skills. It takes Claude to another level. Even inside Claude Code, Skills is the best tool for packaging custom instructions. I prefer making Skills over subagents because they work in the main chat context and can bundle instructions with multiple data files and code. My main observation: Managing AI effectively will require dedicated operations people, because even if you create Skills, you need someone monitoring their performance and tweaking them based on usage to optimize value across team members.</p>
<h2><strong>Where thereвҖҷs a skill, thereвҖҷs a way</strong></h2>
<p>Skills deliver on their technical promise. Progressive disclosure (that bookshelf browsing we mentioned earlier) works. The architecture is sound. The skill creator lowers the technical barrier. But it's not a "set it and forget it" systemвҖ”it requires prompt engineering skills to deploy effectively.</p>
<p>For AI-fluent individuals who understand how to develop descriptions Claude can parse, Skills are worth building now. Just don't assume the skill creator will write perfect descriptions for you on the first goвҖ”review and sharpen them yourself.</p>
<p>For organizations trying to deploy institutional knowledge at scale, the challenges are substantial but solvable. The manual sharing problem needs the promised infrastructure, and the skill description problem requires either training or dedicated support from someone who understands both the domain and how to write for AI.</p>
<p>For everyone else, keep watching. When Skills matures into what it promisesвҖ”a truly automatic system that makes Claude work your way without requiring prompt engineering expertiseвҖ”it'll be powerful. The foundation is already solid. The individual productivity gains are real. The organizational deployment just needs a little time to cook. </p>
