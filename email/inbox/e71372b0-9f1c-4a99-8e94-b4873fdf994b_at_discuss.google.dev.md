    [How to update related parent/child records when a bridging table’s status changes?][6]

      Best pattern here is to let the Prospects (bridging) table drive the update via an Automation Bot: 


Trigger: On update of [Status] in Prospects. 


Process step: Look up the mapping table to translate the new status into the correct Client/Deal statuses. 


Action: Use “Execute an action on a set of rows” targeting Table A and Table B. The key is to filter those sets with refs from the updated Prospect row ([Client] and [Deal]), so only the related rows update. 


Passing values: In the called action on Client/Deal, set the [Status] to something like: 
ANY(SELECT(Mapping[ClientStatus], [ProspectStatus]=[_THISROW].[Status]))  


(same for Deal). 


That way, the bridging row’s status flows into the update, and you’re only touching the directly linked Client/Deal, not all others. This keeps the logic centralized and avoids conflicts when multiple prospects exist.
