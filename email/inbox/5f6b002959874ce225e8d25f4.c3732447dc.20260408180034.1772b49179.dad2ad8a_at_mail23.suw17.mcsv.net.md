And today, we have one — Semaphore UI v2.17 is out! This release focuses on improving daily workflows, while also introducing stronger access control and high availability for production environments.

Here’s what’s new:
https://semaphoreui.us2.list-manage.com/track/click?u=5f6b002959874ce225e8d25f4&id=03d2726322&e=c3732447dc

One-time scheduled tasks

Schedule a task to run at a specific date and time, without creating a recurring schedule. Perfect for one-off releases, migrations, maintenance windows, or late-night deployments.

Customizable views

Adjust the interface to better match your workflow. Faster navigation and easier access to what matters most.

Visual Variables & CLI

Reorder variables visually instead of editing them manually — especially helpful for complex playbooks.

Read Secrets from Raw Files

Source secret values directly from filesystem paths at task runtime. Works natively with Kubernetes Secrets and CSI drivers (Vault, AWS Secrets Manager, Azure Key Vault).

Force Stop All Tasks

Immediately terminate all running and queued tasks, bypassing graceful shutdown. An emergency escape hatch for runaway playbooks or before taking a node offline.

For teams running Semaphore UI in production, v2.17 also introduces:

Custom Roles & Extended RBAC

Define custom permission sets beyond the built-in roles — giving teams precise control over who can view, run, or manage resources across projects.

Active-Active High Availability

Run Semaphore in distributed mode with multiple active nodes — eliminating single points of failure and improving resilience.

BoltDB → SQL Migration Tool

v2.17 ships a built-in migration tool to move all your data — projects, templates, tasks, users — to SQLite, MySQL, or PostgreSQL. Docker users can trigger it automatically via SEMAPHORE_MIGRATE_FROM_BOLTDB.

Remote Secret Storage Sync

When secrets rotate in your external backend (Devolutions Server and others), keep Semaphore in sync with one click — or automate it via API as part of your CI/CD pipeline. No more manual updates, no stale credentials causing failed deployments.
