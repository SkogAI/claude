Here's what shipped this week, plus a few updates from the past few weeks worth knowing about. Let's get into it.




API v3.1: Stay on the latest tools, automatically



Most of you probably haven't thought about toolkit versioning, and that's the problem.
 
When you call the v3 tool APIs without specifying a version, you get toolkit version 00000000_00, the very first version of that
toolkit. Every improvement we've shipped since then (new tools, bug fixes, schema updates) lives in later versions. If you've
never explicitly passed toolkit_versions: "latest", your agents are running on tool definitions that may be months old.
 
What v3.1 changes. In v3.1, tool endpoints default to the latest toolkit version automatically. No flag to pass, no version to
remember. Fixes and new tools reach your agents as soon as they ship.
 
Why you should migrate now. Everything we're shipping from here on out lands in v3.1 first, including upcoming Usage APIs that
will only be available on v3.1. If you're still on v3, you'll start missing things.

How to move over:
 * Using the SDK? Nothing to do. We'll adopt v3.1 in an upcoming SDK release.

 * Calling the API directly? Swap /api/v3/ for /api/v3.1/ in your tool endpoint URLs. If you were already passing version:
   "latest", you can drop the parameter entirely.

 * Relying on the pinned 00000000_00 default? Move to /api/v3.1/ and explicitly pass version=00000000_00 wherever you want to stay
   pinned. But really, consider moving to latest.

 
Only tool endpoints change. /auth_configs, /connected_accounts, /triggers, /toolkits all behave identically between v3 and v3.1.

Multiple Connected Accounts


Real users don't have one Gmail. They have a work Gmail, a personal Gmail, and the one they use for side projects. Same story for
GitHub orgs, Slack workspaces, Notion accounts. Until now, building an agent that handled this meant building in sessions.
 
Multiple Connected Accounts is now generally available for developers building on the Composio SDK and for anyone using Composio
For You directly.
 
If you're building agents: Your users can now connect more than one account for the same toolkit within a single session. Sessions
default to the most recently connected account, or you can pin a specific one per session. Each connection can carry a
human-readable alias like work-gmail or personal-github so your agent (and your users) don't have to think in IDs.
 
If you're using Composio For You: Go add your second Gmail, your other GitHub org, your side-project Slack. Your agent will pick
the right one based on context, or you can tell it which to use.

Project security settings worth turning ON



Your Project Configuration page has four security settings that protect your agents and your users' data in production. These are
auto-enabled for recently created projects, but if you've been with us for a while, they may not be on yet, so worth checking:


 * Two-Factor Authentication: requires user id and connected account IDs for all actions

 * Mask Connected Account Secrets: hides sensitive information in connected account details

 * Require API Key for MCP: requires API key authentication for MCP connections
   Enable Link Flow for Managed Auth: uses Link Auth flow for OAuth connections with Composio-managed credentials
   
   



Head to Project Configuration and turn these on for your production projects. They're the baseline we'd recommend for anything
handling real user data.

Go to Project Settings → [https://platform.composio.dev/]

Also worth knowing


A few updates from the last few weeks worth paying attention to. Some are quality-of-life wins, some are things you'll want to act
on.



 * Bring your own tokens (bearer token auth + credential patching). If you've already authenticated your users on your end and
   want to use those tokens with Composio sessions, you can now do that directly. All OAuth2 toolkits: Gmail, GitHub, Slack,
   Google Docs, and the rest, accept BEARER_TOKEN as an auth scheme. Existing OAuth connections are unaffected. When you refresh a
   token on your end, the new PATCH /api/v3.1/connected_accounts/{id} endpoint lets you push the update without recreating the
   connection. Currently available via API and SDK only, not the dashboard UI yet. Changelog →
   [https://docs.composio.dev/docs/changelog/2026/04/07]

 * Composio sessions are where the platform lives now. If you installed Composio a while back and stepped away, the biggest thing
   to catch up on is sessions. Earlier this year we took what was previously Tool Router and made it the default way to work with
   tools. Sessions solve the problem most agent builders hit at some point: context management. Instead of stuffing hundreds of
   tool definitions into your agent's context window, a session gives your agent a small set of meta tools that discover,
   authenticate, and execute across 1000+ apps at runtime. Cleaner context, faster agents, less plumbing. How Composio works →
   [https://docs.composio.dev/docs/how-composio-works]

 * Dedicated MCP Servers are now legacy. Sessions is the recommended path for new MCP configs, it's more flexible, supports tool
   search and planning, and is where all new MCP capabilities land. Existing Dedicated MCP Servers continue to work; we recommend
   new configs use Composio For You → [https://dashboard.composio.dev/~/org/connect]
