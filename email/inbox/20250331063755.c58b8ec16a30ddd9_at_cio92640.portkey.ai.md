Hey Emil,

LLMs are getting eerily good at sounding right while being utterly wrong. They can return a perfectly formatted 200
response, that hallucinates facts, shows bias, or violates privacy – all while seeming perfectly legitimate.
It's becoming a serious risk for businesses.

That’s why we brought guardrails inside the gateway. They enforce behaviour in real-time, checking both
inputs AND outputs on the fly making sure your LLMs never behave improperly!

Guardrail Flow Diagram

What you get out of the box :

* Over 20 built-in checks, from simple regex matching to advanced LLM-based gibberish detection
* Custom guardrail integration, so you can keep using your existing checks
* Partnerships with top guardrail providers like Aporia, Pillar, and Patronus AI
* Flexible actions to deny risky requests, add automatic feedback or fallback to safer alternatives

Here's how you can start leveraging guardrails today:

* Create a guardrail on Portkey
* Choose whether to fail the request if guardrail fails & define other actions
* Add the guardrial to a config, and attach the config to a request

{
"retry": {
"attempts": 3
},
"cache": {
"mode": "simple"
},
"virtual_key":"openai-xxx",
"before_request_hooks": [{
"id": "input-guardrail-id-xx"
}],
"after_request_hooks": [{
"id": "output-guardrail-id-xx"
}]
}
