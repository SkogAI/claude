✨ Now Available: Zitadel Version 4 RC ✨

Here are some exciting new details and updates from the Zitadel team for July:

- The new resource-based API is a fresh take on a feature that offers flexibility when performing resource management tasks (i.e. user creation, instance creation, permission management) that echo across the Zitadel platform.

The API and its services use a resource-oriented design. This means that the API is designed around resources, which are the key entities in the system. Each resource has a unique identifier and a set of properties that describe the resource. The entire lifecycle of a resource can be managed using the API. This release will only support GRPC with connectRPC (https://djX2BV04.eu1.hubspotlinks.com/Ctc/5H+113/djX2BV04/VVppRY4BBqWZW1gq55y87m5X8W91lZlN5yMlPCN5yXtkR3pyd0W7lCdLW6lZ3ksW727P5Z5pg6G4W2y8lWf1-bt6mW8gKM5R8l-7SCW1B9b_b5xng8LW7C_5Kx9lND4bW67hKcT4kBqq5W70YLQD8XL9fGW4-pzd_5tyl25W4GvHbJ55S-9hW4xH2Z42RpqdzW6tVvYd472XvYW5fjCxR3bTKwgN2wDH0cZWVLqW7LNNns1g0z78W9g60H85yZZVWW71nx5M17tldyW31VdwS1XXlMfW60l74K1lx8qLW78bDdJ1XW_z1W2PkxP35hXvK8W6-YZq918jlCcN6-Wk77n4BRzMRw6kQzYHJWW3bgB932lVH4ldClGq004 ) and remove support for OpenAPI 2.0.

With this feature, all resources now have their own supporting service. This limits impact to other aspects of the business as it runs in parallel to produce a separation of concern. Whereas the  old API (https://djX2BV04.eu1.hubspotlinks.com/Ctc/5H+113/djX2BV04/VVppRY4BBqWZW1gq55y87m5X8W91lZlN5yMlPCN5yXtl63pyd0W7Y8-PT6lZ3nvW61_JLL6WfgKdW2V3lq_2JhRJ8VhXqdb3wzPgxW8Kc2-p5fcp0wW386RK_4zWxBtW7wW1Sg1y9pSvW34sXWs97jWm6W6RPCMC75rT3_V81jTs2dyd4hN8gwmc4j5J5YW61FrgK13QNVZW6kmnDF1rkLFhW5SV-l84bpqjSW27J6kr3NBq0TVs87L15WJQGcW7zMcWY26lL2JW4-J5z226n0T9W2SPS3R1Hrpy-W6c2_r317NGJ-W6h8dnP4MgJ7zW5pQPxW3BHhl3W1_-cw049-lt-W4dVF5525-4wRW6wRT5s2wkh1xV8nLz77jjhDRW69v9HP2fZ56vf9bYbj804 ) required having the full scope when updating the resource, this new resource-based API no longer hinders such workflows as you can partially update resources as needed.

With the goal to pair users together based on specific criteria or attributes, there currently is not a traditional Group feature, there are two potential workflows for accomplishing the same result.

1st Workflow: As a Zitadel admin, you have the ability to add metadata with the group name to the User to generate the association.

2nd Workflow: There is also the function of leveraging the roles within a project such as "group1" or "group2", however that would be limited to the context of that specific project.

User groups are in development now and will be available later (https://djX2BV04.eu1.hubspotlinks.com/Ctc/5H+113/djX2BV04/VVppRY4BBqWZW1gq55y87m5X8W91lZlN5yMlPCN5yXtl63pyd0W7Y8-PT6lZ3pTW5XD4588P4H-5W278bFh3JtfSFW5qY6r46Zd3KZV3TYFj3XL5zxW46w9XM4sxrGMW6n_QBP6S1NgsW8M-WFd6dFvcsW8pQ9qs5gfkgrN5BMWhFYBBRYW33LQbk4qDCp7W8RZJgC71wvczW8hzNSd3fqYq2W4RZd3N1vwL6yW2xCt_q1b6LQpW19ypsq45GZfNW7sk60G5JnPlGW2LpTzL3vk98nW8zd7RM8mY3SNW8myjqP1mdfQsW7ZsFjS6xZkN1W8Wb_bl90K1CmW1m8Yr885WpHyMrcpPL8GyYNW6c1rJR8nhKfgW7n6BPZ8KTctgW1QmrL81jRCl8f9d_dK-04 ) in 2025!
